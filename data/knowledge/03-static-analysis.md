# IDA / Ghidra 静态分析工作流

## 打开样本后先做什么

1. 确认架构、入口点、编译器和是否有壳。
2. 浏览导入表、字符串、资源、异常表、初始化函数。
3. 从明显输出字符串交叉引用到判定逻辑。
4. 给关键函数和变量命名。
5. 对照汇编修正反编译器误判的类型、数组和结构体。

## 字符串策略

优先搜索：

- `flag`
- `key`
- `wrong`
- `correct`
- `success`
- `input`
- `usage`
- `%s`
- `{`
- Base64 字符表或十六进制表

没有明文字符串时，检查是否存在运行时解密：入口附近或主逻辑前可能有 XOR/ROL/ADD 循环。

## 交叉引用

常用路径：

```text
字符串 -> xref -> 输出函数附近 -> 条件分支 -> 比较函数 -> 输入变换
导入 API -> xref -> 行为函数 -> 参数来源 -> 关键数据
全局表 -> xref -> 循环 -> 变换逻辑
```

只看一个引用容易误判，至少确认调用链和运行时是否真的经过。

## 伪代码使用原则

反编译器是阅读辅助，不是事实来源。以下情况必须回看汇编：

- 变量类型明显异常。
- 数组下标和指针运算混在一起。
- 出现奇怪的有符号/无符号比较。
- switch 被还原得很乱。
- SIMD、位运算、异常处理、setjmp/longjmp。

## 命名建议

命名应该表达证据，而不是猜测：

| 不推荐 | 推荐 |
| --- | --- |
| `check_flag` | `compare_transformed_input` |
| `decrypt` | `xor_decode_resource` |
| `key` | `global_xor_table_32` |
| `sub_401000` | `read_user_input` |

分析推进后再把保守命名改成更明确的名字。

## 结构体和数组

当看到固定 stride 访问时，优先建立结构体：

```text
base + i * 0x18 + 0x00
base + i * 0x18 + 0x08
base + i * 0x18 + 0x10
```

这通常比把每个字段当成独立全局变量更清晰。

## 脚本化

适合写脚本的场景：

- 批量重命名函数。
- dump 解密后的字符串。
- 提取只读表、S-box、密文。
- 根据调试日志复现变换。

脚本输出要保存到独立文件，避免污染原始样本。

## 静态分析完成标准

至少能回答：

1. 输入从哪里进入？
2. 经过哪些确定的变换？
3. 最终与什么比较？
4. 失败和成功分支在哪里？
5. 是否能写脚本生成正确输入？
